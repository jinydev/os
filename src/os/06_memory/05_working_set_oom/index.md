# 5. 메모리 붕괴: 스래싱(Thrashing)과 시스템 최후의 방어벽 OOM Killer

CPU를 100% 혹사시켜 처리량을 늘리겠다고 메모리 한도에 부딪히며 프로세스 수량을 끝없이 늘리다 보면 결국 공학적 재앙이 터집니다.

## 스래싱(Thrashing) 붕괴와 워킹 셋(Working Set) 제한
어느 프로세스든 각자의 업무를 무사히 돌리기 위해 자신만의 최소 집중 활동 메모리 구역인 `Working Set(작업 공간)`이 보장되어야 합니다.
그런데 실행 중인 프로그램이 너무 많아져 램 공간이 극도로 모자라다 보니, 1초가 멀다 하고 서로 남의 공간을 뺏고(`Swap Out`), 내 것을 다시 하드디스크에서 불러오는(`Swap In`) 바보 같은 I/O 헛발질만 일삼게 됩니다. 

정작 CPU 연산 코어는 "데이터가 아직 안 왔으니 난 디스크만 기다릴게..." 라며 놀아버리는 극한의 비효율 병목 현상, 가동률이 바닥으로 수직 추락하는 이것이 **스래싱(Thrashing)** 붕괴입니다.

![스래싱 현상 그래프](./img/os11_thrashing.svg)

따라서 현대 OS는 현재 집중 사용되는 '자주 참조된 페이지 집합(Working Set)' 크기의 총합을 측정하여, 남은 물리적 RAM 공간이 이 임계치보다 쪼들리게 되면 차라리 아예 신규 앱의 프로세스 로딩을 사전에 차단(Suspend)해 버립니다. 

<br>

## 🔪 리눅스의 스나이퍼: OOM Killer

모든 RAM과 최후의 보루인 하드디스크 스왑(SWAP) 파티션마저 고갈되어 시스템 전체 패닉(Panic)이 도래하기 직전, 무자비한 Linux 커널은 최후통첩 스나이퍼를 작동시킵니다. 바로 **OOM (Out-of-Memory) Killer**입니다.

리눅스 커널은 상주하는 프로세스들마다 동적으로 `badness()`라는 페널티 점수표를 매깁니다. 이 점수를 기반으로 우선순위(nice)가 낮고 비정상적으로 뚱뚱하게 메모리를 차지하고 있는(메모리 릭 예상) 최적의 타겟 프로세스 1개만을 포착하여, OS의 가장 높은 권한으로 해당 프로세스에 강제 총살(SIGKILL `-9`) 인장을 날려 메모리를 비강제로 뜯어 탈환합니다. 

여러분이 서비스하는 사내의 Java Spring Boot나 Python 데이터베이스 서버가 도무지 아무 에러 로그(Exception)도 남기지 않고 순식간에 급사했다면, 즉시 커널의 깊은 로그 명령어인 `dmesg`를 뒤져 리눅스가 쏜 OOM Killer 발생 총살 이력을 찾아내야만 합니다.
