# 2. 모든 것의 목차: UNIX i-node (Index Node)

사용자는 "a.txt", "logo.png"와 같은 친숙한 인간의 '이름'으로 파일을 찾지만, 커널 백엔드에게 그건 허상일 뿐입니다. 리눅스 시스템은 무조건 고유한 일련번호인 **i-node 번호**만으로 파일을 판별합니다. 여러분이 탐색하는 '디렉토리(폴더)' 객체는 대단한 것이 아니라, 그저 "문자열 파일명 -> inode 번호"를 세로로 나열해 놓은 명부 테이블 텍스트 쪼가리에 지나지 않습니다.

![UNIX i-node 아키텍처](./img/os12_inode_architecture.svg)

## 팽창하는 인덱싱의 기적
i-node 맵핑 체계의 위대한 점은 **저장할 파일 용량 팽창에 따라 탐색 깊이가 트리 구조로 동적으로 달라진다**는 것입니다.

작은 텍스트 파일은 141바이트의 i-node 내부에 배정된 **직접 블록(Direct Block)** 포인터만 읽어 즉시 RAM으로 뽑아냅니다. 하지만 넷플릭스 영화 파일 용량이 수기가바이트, 배포 이미지가 거대한 테라바이트 급으로 커지면, i-node 내부 포인터는 포인터의 포인터를 참조하는 **일중, 이중, 삼중 간접 블록(Triple Indirect Block)** 체계를 다단계로 확장 작동시켜 디스크 클러스터 파티션 반경 안을 거미줄처럼 무한 동적 할당해 기적적인 트리를 완성해 냅니다.

> [!WARNING] **(실전 트러블슈팅) inode 고갈 이슈**
> 백엔드 서버 디스크 전체 물리 용량(`df -h`)은 500GB나 텅텅 남았는데 갑자기 데이터베이스나 Nginx에서 "No space left on device" 에러가 뜬다면? 
> 수십 킬로바이트짜리 아주 작은 임시 세션 텍스트 파일들이 수백만 개 폭증하여, OS가 파일 생성마다 소모하는 **총 i-node 개수 한계점**에 다다른 것입니다! (이럴 땐 `df -i` 캐피시티로 색인 노드 한도 모니터링이 필수입니다.)
