# 1. 파일과 아키텍처: 디렉터리 시스템과 링크

초창기 유닉스의 디렉터리는 가계도와 같은 단순한 트리(Tree) 계층 구조로 설계되었으나, 똑같은 타겟 파일(ex: `shared.dll`이나 공용 환경설정 스크립트)을 여러 유저 폴더에 각각 중복해서 복사해 두어야 하는 최악의 용량 낭비 파편화 한계가 존재했습니다. 
이를 타파한 획기적 디자인이 모든 파일과 폴더를 거미줄처럼 그물망으로 엮어 어디서든 참조할 수 있게 만든 **비순환 그래프 레이어(Acyclic Graph)** 구조입니다.

![디렉터리 구조 비교](./img/os12_directory_structure.svg)

## 공용(Sharing) 참조 체제의 양대 산맥: OS 링크(Link)
파일을 물리적 복사본 없이 논리적으로 동시 다발 공유하는 체계를 링크(Link)라 부릅니다.

* **하드 링크 (Hard Link / `ln`)**: 블록 데이터가 존재하는 원본 실체를 향해 "새로운 폴더 디렉터리의 이름표(Pointer)"만을 한 개 더 창조합니다. A를 수정하면 B가 100% 동일하게 번경되며, 시스템은 레퍼런스 카운팅(Reference Counting) 기법을 사용해 원본을 가리키는 이름표들을 0이 될 때까지 다 지우기 전엔 절대로 실제 데이터를 디스크에서 삭제시키지 않습니다.
* **심볼릭 링크 (Soft Link / `ln -s`)**: 현대 윈도우의 '바로가기' 아이콘과 동일합니다. 데이터 원점 실체를 가리키는 것이 아니라 방황하는 껍데기뿐인 타겟 원본 '경로 문자열' 텍스트만 저장하여 가리키므로, 사용자가 타겟 원본 파일을 지워버리면 소프트 링크는 방향성을 잃은 쓸모없는 빈껍데기 깡통(Broken Link)이 되어버립니다.
