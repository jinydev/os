# 4. 문맥교환의 한계, 그리고 멀티 스레드(Thread) 진화

각 프로세스는 무거운 전용 PCB 덤프 자원과 완벽하게 격리된 거대 4대 메모리를 갖지만, 이 때문에 프로세스 간 통신 및 컨텍스트 스위칭 전환 비용이 너무 높았습니다. 

이러한 무시무시한 오버헤드 병목 현상 해소를 위해 현대 렉처는 데이터(Data/Heap) 블록은 서로 뚫어놓고 공유한 채, 독립적인 함수 호출에 관련된 **스택(Stack) 전용 메모리 공간과 PC 레지스터만 별도로 추적하는 가장 가벼운 개체**, 스레드(Thread) 통제를 학습합니다. 

![단일스레드 다중스레드 대비](./img/os04_12.svg)

스레드 단위마다 부여되는 독자적 TCB(Thread Control Block)를 활용하면, 한 크롬 탭에서는 브라우저 모니터 렌더링을 띄우고 다른 뒤쪽 백그라운드 스레드에서는 무거운 네트워크 통신을 진행하는 완벽한 **비동기 렌더링(Asynchronous Rendering)**이 구현됩니다. 하지만 Data를 양쪽이 공유한다는 것은 결국 두 스레드가 같은 메모리를 덮어쓰려 충돌하는 **경쟁 상태(Race condition) 동기화 오류**라는 지옥에 문을 두드리는 일과도 같습니다.

## 사용자 수준 1:N 맵핑 vs 커널 1:1 맵핑 매커니즘
유저스페이스 라이브러리가 임의로 TCB를 제어하는 N:1 스레드는 커널 트랩이 없으므로 속도는 극강을 찍지만, 한 스레드의 I/O(디스크 쓰기 등)를 위해 커널 차원에서 차단(Block)되면 전체 스레드 군단이 모조리 멈춰버리는 **블로킹 종속 이슈**에 취약합니다. 

결국 현대의 모든 리눅스/윈도우 멀티코어 환경은 커널 코어 단위로 1:1 매핑되어 스폰되는 무거운, 그러나 절대적으로 안전한 **커널단 1:1 대응 NPTL 스레드 모델**을 기초 아키텍처로 사용합니다.

## [실전 심화] 스레드의 분쇄와 코루틴(비동기 논블로킹 I/O)

하지만 세계에서 가장 널리 쓰이는 백엔드 툴셋 Python은, 이 스레드 공유 메모리의 레퍼런스 카운팅 붕괴를 막고자 전체 인터프리터 자체에 방공호 락을 걸어버리는 **GIL(Global Interpreter Lock)**을 강제합니다! 아무리 100 코어 서버 머신이라 할지라도 파이썬 코드 스레드는 코어 개수에 관계없이 동시에 단 1개만 실행됩니다.

그에 대한 해답으로 등장한 것이 Node.js와 Python의 거대 메가 트렌드인 `asyncio` 패턴, 바로 **코루틴(Coroutine) 논블로킹 아키텍처**입니다. 무거운 커널 스레드를 고작 1개만 돌리되, I/O 대기 시간에 재빨리 이벤트 루프(Event Loop)를 통해 유저-레벨 함수 제어권을 상호 반환(Switching / `await`)하여 초당 수만 건의 쿼리를 딜레이 없이 단일 코어 하나로 가볍게 넘겨버리는 극한 효율의 프로그래밍 기법입니다.
